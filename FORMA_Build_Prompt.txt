DESIGN SYSTEM





FORMA Design Language



FORMA is built around a single aesthetic principle: restrained luxury. The design must feel like a premium editorial product — warm cream surfaces, deep wine accents, refined serif typography, and motion that breathes rather than performs.







Do not allow the AI to use Inter, Roboto, or any system sans-serif font. Do not allow purple gradients, blue buttons, or card shadows that look like Material Design. If the AI deviates from the design system below, correct it immediately before continuing.

Permitted Animations



All animations must feel like a breath — not a performance. The following are the only animation styles permitted in FORMA:











A

Entrance Fade-Up — Elements enter from 18px below with opacity 0 → 1, duration 0.6s, ease: cubic-bezier(0.22, 1, 0.36, 1), staggered by 80ms per child.











B

Underline Reveal — On hover over links and nav items, a 1px wine-colored underline grows from left to right using scaleX(0) → scaleX(1) with transform-origin: left, duration 0.3s ease-out.











C

Button Press — Buttons scale down to 0.97 on mousedown, spring back on release. Duration 150ms. No glow, no shadow pulse.











D

Panel Slide-In — The properties sidebar slides in from the right: translateX(100%) → translateX(0), duration 0.35s, ease: cubic-bezier(0.22, 1, 0.36, 1).











E

Section Highlight Pulse — When a section is selected in the builder, its border pulses once: opacity 0 → 1 → 0.6 → 1, duration 0.5s. No looping.











F

Drag Ghost — While dragging a section, the dragged item renders at 70% opacity with a subtle wine border. The drop target shows a 2px wine line.











G

Skeleton Loader — Before content loads, show a shimmer animation: linear-gradient moving from left to right in wine-muted tones. No spinners.











No other animations are permitted. No bouncing, no rotating loaders, no particle effects, no confetti. If the AI adds any animation not listed above, instruct it to remove it.

PHASE 2





Dynamic JSON Rendering & Zustand Store



This phase converts the static sections into a live, data-driven builder. The page is no longer hardcoded — it renders from a JSON structure stored in Zustand.







Prompt 2.1 — Build the Zustand Store







Build the Zustand store at /src/store/builderStore.ts







The store must manage the following state:



- sections: SectionProps[]  (starts with DEFAULT_PAGE)



- selectedId: string | null



- isDragging: boolean



- history: SectionProps[][]  (for undo/redo)



- historyIndex: number







The store must expose the following actions:







addSection(type: SectionType) — adds a new section at the end



  with a generated ID and default props for that type







deleteSection(id: string) — removes section from array







selectSection(id: string | null) — sets selectedId







updateSectionProp(id, key, value) — updates a single prop



  on the specified section without mutating others







reorderSections(newOrder: SectionProps[]) — replaces sections array



  after a drag-and-drop reorder







undo() — moves historyIndex back by 1 and restores that snapshot







redo() — moves historyIndex forward by 1 and restores that snapshot







Every mutation except selectSection must push the current state



to history before applying the change (max history depth: 30).







Use immer middleware for clean state mutations.

- Pass the section&apos;s id and props down







Use a switch statement or lookup map to resolve section type → component.







The Canvas itself:



- Takes the full available height of the editor area



- Background: var(--color-cream)



- Overflow: auto (scrollable vertically)

When no sections exist, show a centered empty state:
